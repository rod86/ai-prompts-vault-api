---
name: write-operation-conventions
description: Design conventions for create/update (write) operations, established in 005-create-prompt
metadata:
  type: feedback
---

- **Existence-check business errors on a body field are 400, not 404** — a
  create/update operation validating a referenced-by-id field (e.g.
  `category_id` on `POST /prompts`) throws a domain `*NotFoundError` mapped to
  `400 { error: message }`, deliberately different from a URL-addressed
  resource's `404` (`GET /prompts/:id`'s `PromptNotFoundError`). Log the
  rationale explicitly in spec.md's Decisions log each time: "business-rule
  check on a body field of a write, not a missing URL-addressed resource."
  Expect to make this same 400-vs-404 call again for any future create/update
  feature that references another entity by id.
- **Format validation vs existence validation are two separate spec items**,
  even for the same field — a V# (Zod-level shape check, e.g.
  `z.string().uuid()`) for "is this even a plausible id", and a separate E#
  (thrown domain error) for "does it refer to something real". Don't collapse
  them into one AC.
- **Deliberate reversal of the "opaque filter value" convention** from
  list/read features (see [[list-read-conventions]]): reads treat a
  non-matching filter as a normal empty result with no format check; writes
  validate the referenced id's format AND existence, because a bad reference
  must reject the write rather than silently produce nothing. State this
  divergence explicitly in the spec's Decisions log so it doesn't read as an
  inconsistency.
- **Use-case purity for writes**: per `coding-style`'s "no I/O/clock/randomness
  in domain functions" rule, a `Create*UseCase` takes `id`/`createdAt`/
  `updatedAt` as plain fields on its `Query`, generated by the **handler**
  (`randomUUID()` + one shared `new Date()` call so `createdAt === updatedAt`
  at creation), never inside the use case itself.
- **No re-fetch after insert**: a create use case assembles and returns the
  full entity in-memory from the query fields + the already-looked-up related
  entity (e.g. the looked-up `category`), then calls the repository's
  `create()` (returns `void`) — it does not re-query the database to build
  its return value.
- **Repository `create(entity)` takes the full domain entity**, not a flat
  "model"/row-shaped type — the adapter itself destructures nested fields
  (e.g. `entity.category.id` for the FK column) at the point of the
  `db.insert(...).values({...})` call. Mirrors the existing read-side pattern
  where nested shape only exists at the domain layer, not in the DB row.

**Why:** these were explicit, deliberate decisions given for 005 (the first
write feature) diverging from established read-side conventions; future
writes should follow the same pattern rather than re-deriving it.
**How to apply:** default to these for any new create/update feature; call
out the divergence from [[list-read-conventions]] explicitly in the spec's
Decisions log rather than silently.
