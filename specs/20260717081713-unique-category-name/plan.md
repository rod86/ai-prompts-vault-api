# Plan: Unique category names
Spec: specs/20260717081713-unique-category-name/spec.md

## 1. Approach
Add a case-insensitive uniqueness guarantee to the `name` column of the
`prompt_categories` table by declaring a unique index on `lower(name)` in the
centralized Drizzle schema, then generating and applying the corresponding SQL
migration. There is no application, domain, or HTTP change: categories have no
write endpoint (only `findAll`/`findById`), so the rule lives entirely at the
storage boundary.

Per the user's instruction, **no new automated test is added** — the change is a
declarative schema/migration, not application logic, so it falls under the SDD
"logic-less change" exception (proven by a clean apply/type-check rather than a
test). AC1–AC3 are verified by generating the migration, confirming it emits
`CREATE UNIQUE INDEX … (lower(name))`, and applying it cleanly against the seed
data. The case-insensitive rejection (AC2) is additionally demonstrated with a
**manual insert check** after the migration is applied: insert `Category1`
(succeeds), then insert `category1` (must be rejected by the index), then delete
both rows so the check leaves no residue. One supporting test-infrastructure
change is still required: hardening the category model factory so the **existing**
integration suite — which inserts random-named categories — cannot collide under
the new index and flake.

## 4. Data & persistence
**Table**: `prompt_categories`
| Column | Type | Nullable | Default | Constraints | Description |
|--|--|--|--|--|--|
| id | UUID | No | — | Primary key | Surrogate identifier (unchanged) |
| name | text | No | — | Unique on `lower(name)` | Category label; now case-insensitively unique (V1) |

- Migration: `CREATE UNIQUE INDEX "prompt_categories_name_unique" ON "prompt_categories" USING btree (lower("name"));`, generated by `npx drizzle-kit generate` from the schema change (drizzle-kit discovers tables via `src/config/drizzle/schema.ts`).
- Rollback: `DROP INDEX "prompt_categories_name_unique";`.
- Existing data: the seed (`drizzle/0001_seed_prompt_categories.sql`) holds 11 categories whose names are distinct even ignoring casing, so the index builds against current data without conflict (see assumption 2).
- Mapping: domain `name` ↔ column `name` (no rename).

Schema change shape (declaration only, no logic):
```ts
export const promptCategories = pgTable(
    'prompt_categories',
    {
        id: uuid('id').primaryKey(),
        name: text('name').notNull(),
    },
    (table) => [uniqueIndex('prompt_categories_name_unique').on(sql`lower(${table.name})`)],
);
```

## 7. Assumptions & risks
Assumptions:
1. No client-facing endpoint creates, renames, or deletes categories today
   (confirmed: only `ListPromptCategoriesUseCase` plus `findAll`/`findById` exist),
   so the guarantee is storage-level only and needs no domain error type or handler
   mapping — consequence if wrong: a future write endpoint would need a domain error
   surfacing E1 as an API response, which is a separate spec.
2. The current seed data has case-insensitively distinct names — consequence if
   wrong: the unique index fails to build until the duplicates are resolved.
3. The migration is applied manually with `npm run db:migrate` before the suite runs
   (project convention; the app does not migrate on startup) — consequence if wrong:
   the new constraint test fails because the index is absent.

Risks:
| # | Risk | Likelihood | Impact | Mitigation |
|--|--|--|--|--|
| R1 | `PromptCategoryModelFactory` draws names from `faker.commerce.department()` (a small pool); random category inserts could collide under the new index and cause intermittent failures | med | flaky integration tests across the suite | Harden the factory to append a unique suffix so generated names never collide (T1) |

## 8. Edge cases
| Case | Input / state | Expected behavior | Covers |
|--|--|--|--|
| Duplicate exact name | Insert "Productivity", then "Productivity" again | Second insert rejected; one row remains | AC1 |
| Duplicate differing only in case | Insert "Productivity", then "productivity" | Second insert rejected; one row remains | AC2 |
| Distinct name | Insert "Productivity", then a different name | Both stored | AC3 |
| Index build over existing seed | Apply migration with the 11 seed rows present | Index builds; all rows retained | none (assumption 2) |
| Manual post-migration check | Insert `Category1`, then `category1`, both against `prompt_categories` | First insert succeeds; second is rejected by the `lower(name)` index; delete both rows afterward | AC1, AC2 |
