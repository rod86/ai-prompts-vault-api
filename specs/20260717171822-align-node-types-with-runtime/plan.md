# Plan: Align development type definitions with the supported runtime
Spec: specs/20260717171822-align-node-types-with-runtime/spec.md

## 1. Approach
`package.json` sets `engines.node` to `>=24` (and the README pins a v24 runtime), but the
development type-definitions package that types the codebase, `@types/node`, is pinned to
`^20.14.10` — a major version behind the supported runtime (report finding L1). The fix is
a single dependency bump: move `@types/node` to major `24`, resolved to the latest stable
`24.x` at install time (consistent with this project's "install at latest stable" habit),
then reinstall and confirm `npm run typecheck` still passes with no new errors. No source
or configuration code changes are expected — this only realigns the type surface with the
runtime the project already targets.

## 7. Assumptions & risks
Assumptions:
1. `@types/node@24`'s type definitions do not surface new type errors in the existing
   source — consequence if wrong: the bump reveals a real v24 API-typing mismatch that
   must be fixed under this spec (returns to PLANNING if it needs a design decision).
2. "Version 24" means major 24 at latest stable within that major (per the project's
   latest-stable convention), not an exact pin — consequence if wrong: user wants a
   specific patch; adjust the version string.

Risks:
